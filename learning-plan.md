# 开发岗 Agent 一个月学习计划（2026-03-23 至 2026-04-19）

## 适用对象

- 有基础编程能力
- 做过一些 LLM / Web / 小型项目
- 想在 1 个月内做出一个能写进简历、能在面试里讲清楚的 Agent 项目

---

## 本月选定项目

### 项目名称

**DevDocs Agent**

### 项目定位

一个面向开发者的文档与代码库智能助手，支持：

- 对技术文档和仓库说明进行检索问答
- 根据问题自动选择检索、网页搜索或工具调用
- 输出带引用的回答
- 展示基础执行轨迹、耗时和工具调用记录

### 为什么选这个项目

这个项目比“聊天机器人”更像开发岗实际项目，原因有三点：

1. 能覆盖 Agent 开发最核心的工程能力：RAG、Tool Use、Workflow、Memory、日志、评测、部署
2. 和你的现有背景更贴近：你已经做过 Python / Go / Web3 相关项目，开发者文档场景更容易做深
3. 面试可讲性强：为什么不用纯搜索、为什么不用纯 RAG、为什么需要路由和工作流，这些问题都能自然展开

### 技术栈建议

- 后端：FastAPI
- Agent 编排：LangGraph
- 模型接入：OpenAI 兼容接口或其他兼容模型服务
- 检索：FAISS / Chroma（二选一，先求简单可用）
- 数据处理：Markdown / TXT / PDF 解析
- 存储：SQLite 或 PostgreSQL（二选一，先 SQLite）
- 缓存：Redis（可选，最后一周再加）
- 部署：Docker
- 观测：结构化日志 + 简单评测脚本

### 最小功能范围

- `/chat` 问答接口
- `/ingest` 文档入库接口
- 文档切块和向量检索
- 2 个工具：网页搜索、网页内容抓取
- LangGraph 工作流：路由 -> 检索/搜索 -> 生成 -> 校验引用 -> 返回
- 会话级短期记忆
- 日志、错误处理、超时和基础回退
- 20-30 条评测样本
- Docker 启动说明

### 加分项

- 简单前端 Playground
- 仓库级代码文档问答
- prompt / workflow 配置化
- 响应中返回中间步骤摘要

## 每周复盘问题

每周最后一天晚上，固定回答下面 6 个问题：

1. 这个星期我完成了什么？
2. 哪个模块最卡？
3. 当前项目最薄弱的环节是什么？
4. 如果下周只能做一件最重要的事，那是什么？
5. 哪个功能是“锦上添花”，哪个功能是“必须保留”？
6. 这个项目现在能不能拿给别人演示？

---

## 面试导向的重点补充

做项目时要同步准备这些问题的答案：

- 为什么这里需要 Agent，而不是普通聊天机器人？
- 为什么这里需要 RAG，而不是直接把文档塞进 Prompt？
- 为什么需要路由和工作流？
- Tool Calling 失败时怎么处理？
- 记忆怎么做，为什么只做短期记忆？
- 你怎么评估这个系统回答得好不好？
- 如果线上成本太高，你怎么优化？
- 如果延迟太高，你怎么拆解问题？

## 一个月结束后的验收标准

### 项目层面

- 可以本地一键启动
- 至少支持一个真实知识库
- 回答带引用
- 有基本可观测性
- 有基础评测数据

### GitHub 呈现层面

- README 完整
- 有架构图
- 有 demo
- 有清晰的提交记录
- 仓库首页能让人 1 分钟看懂你做了什么

### 能力层面

- 你能独立解释 Agent、RAG、Workflow、Memory、Eval 的关系
- 你能讲清楚项目里每一层为什么这么设计
- 你能回答常见开发岗 Agent 面试问题

## 执行原则

1. 不频繁换题
2. 不同时学太多框架
3. 不只看教程不写代码
4. 不追求功能堆砌，优先把主链路做稳
5. 每周至少有一个可演示版本

---

## 本计划的核心结论

这个月你不需要“学很多”，你需要的是：

- 选一个对开发岗有说服力的 Agent 项目
- 按工程化方式把它做完整
- 让项目、文档、讲法三件事同时成立

如果执行到位，`DevDocs Agent` 足够成为你接下来找 Agent 开发岗时的主项目。
